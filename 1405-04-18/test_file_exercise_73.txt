Displaying the date/time in messages
- در انتهایِ استفادۀ basic میشه چنین چیزی. 
from logging import getLogger, basicConfig, WARNING
logger = getLogger(__name__)
basicConfig(
    filename="logs.md",
    filemode='w', 
    level=WARNING, 
    encoding="utf-8", 
    datefmt="%Y-%m-%dT%I:%M:%S%p",
    format="%(asctime)s:%(name)s:%(message)s"
)
logger.warning('Hi %s %s\n', "erfan", "Heydari",)
logger.critical('This is the End!\n',)

more advanced topics
•	Loggers expose the interface that application code directly uses.
•	Handlers send the log records (created by loggers) to the appropriate destination.
•	Filters provide a finer grained facility for determining which log records to output.
•	Formatters specify the layout of log records in the final output.

 Logging چیست؟
ثبت اتفاقات مهم برنامه برای مانیتورینگ، عیب‌یابی و نگهداری؛ نه صرفاً برای برنامه‌نویس، بلکه برای خودت در آینده، هم‌تیمی‌ها و تیم پشتیبانی.

---

 اجزای اصلی

•	Logger: پیام را دریافت می‌کند.
•	Level: شدت و اهمیت پیام را مشخص می‌کند.
•	Formatter: ظاهر پیام را تعیین می‌کند.
•	Handler: مقصد پیام را مشخص می‌کند (Console، File، ...).

---

 مسیر یک Log

```
Event
  ↓
Logger
  ↓
Level Check
  ↓
Formatter
  ↓
Handler
  ↓
Console / File / ...
```

---

 Level ها

•	DEBUG: اطلاعات جزئی برای توسعه و Debug.
•	INFO: رویدادهای عادی و مهم (مثل Login موفق).
•	WARNING: اتفاق غیرعادی که برنامه هنوز قادر به ادامه است.
•	ERROR: خطا رخ داده، اما برنامه هنوز قابل ادامه است.
•	CRITICAL: خطای بحرانی که ادامه اجرای برنامه یا سرویس را تهدید می‌کند.

( درمورد level ها این نکته جالبه که بسته به اینکه برنامه از لحاظ اجرا در چه مرحله ای قرار داره طبقه بندی میشه. اتفاق غیر عادی ولی خطا نیست، خطایی که برنامه هنوز میتونه ادامه بده (یعنی میدونه چطور باهاش برخورد کنه) و خطای بحرانی که برنامه دیگه نمیتونه ادامه بده. )
---

 Best Practices

•	هر فایل Logger مخصوص خودش را داشته باشد (`getLogger(__name__)`).
•	پیام‌های Log باید شفاف، دارای Context و قابل فهم باشند.
•	از پیام‌های مبهم مثل زیر پرهیز کن:
o	`Done`
o	`Error`
o	`Something happened`

•	همیشه از خودت بپرس:
  > **اگر فقط فایل Log را ببینم، می‌توانم بفهمم چه اتفاقی افتاده؟**

---

 نکات پروژه‌ای

•	یک Logger می‌تواند چند Handler داشته باشد.
•	Handlerها می‌توانند همزمان پیام را به چند مقصد ارسال کنند (مثلاً Console و File).
•	معمولاً در پروژه‌ها هر فایل Logger خودش را دارد و تنظیمات Logging به‌صورت مرکزی انجام می‌شود.

---

 اشتباهاتی که امروز یاد گرفتم

•	بعد از وقوع Error، نباید Log موفقیت ثبت شود.
•	Warning باید علت اتفاق غیرعادی را مشخص کند، نه صرفاً بگوید "اتفاقی افتاد".
•	انتخاب Level باید بر اساس شدت و اهمیت رویداد باشد، نه صرفاً نام آن.
